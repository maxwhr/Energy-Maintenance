## 1. 文档目的

本文档用于明确 Energy-Maintenance 第一版的软件功能需求、业务边界、用户角色、核心数据对象、功能优先级、非功能要求和验收标准，为后续数据库设计、接口开发、前端页面实现、测试报告和部署文档提供依据。

本项目不能仅停留在“知识库问答工具”层面，而应形成面向真实检修作业的闭环系统：

```text
设备台账
    ↓
设备历史故障与维修履历
    ↓
多模态知识库
    ↓
本地/云端大模型检修问答
    ↓
故障辅助诊断
    ↓
标准化作业指引
    ↓
检修任务执行
    ↓
维修记录沉淀
    ↓
知识审核与持续更新
```

本文档重点解决以下问题：

```text
1. 系统第一版到底服务什么场景；
2. 系统必须具备哪些核心功能；
3. 设备台账、维修履历、权限系统是否作为第一版核心功能；
4. 多模态输入和本地小模型如何进入第一版设计；
5. 知识沉淀、审核、人工修正如何形成闭环；
6. 数据库表结构为什么需要增强；
7. 后续 Codex 开发任务应按什么边界推进。
```

## 2. 赛题要求理解

根据赛题描述，本系统需要围绕设备检修场景，解决检修知识分散、检索效率低、作业不规范、经验传承困难等问题。赛题核心要求可以归纳为以下几类：

### 2.1 架构与部署要求

```text
1. 采用 B/S 架构。
2. 软件需要部署在 LoongArch 自主指令集架构 CPU 上。
3. 操作系统为银河麒麟高级服务器操作系统 V11 / V10。
4. 不满足 LoongArch + Kylin 部署要求，将严重影响评分。
5. 开发语言和技术框架不限，但需要具备可落地性。
```

### 2.2 功能要求

```text
1. 支持本地部署大模型服务或云端大模型服务。
2. 具备 PC Web 或 App 可视化界面。
3. 支持多模态知识检索，包括文本、故障图片、设备型号等输入。
4. 支持精准语义检索与跨模态匹配。
5. 支持快速调取检修手册、案例、经验等资源。
6. 支持标准化作业指引。
7. 支持根据设备类型与检修等级推送流程。
8. 支持知识沉淀与更新。
9. 支持一线人员上传检修案例、经验总结。
10. 支持审核后纳入知识体系。
11. 支持人工标注与修正大模型输出结果。
```

### 2.3 非功能要求

```text
1. 界面合理、美观大方。
2. 文档完整规范。
3. 人机交互便捷。
4. 软件具备易用性与稳定性。
5. 演示材料完整，包括文档、源码、安装部署、PPT 和演示视频。
```

### 2.4 对本项目的影响

结合赛题要求，Energy-Maintenance 第一版必须从原本的“知识库问答 + 故障诊断”升级为：

```text
多模态大模型检修作业辅助系统
```

其核心不只是回答问题，而是将设备检修过程中的资料检索、故障诊断、作业指引、任务执行、维修履历和知识沉淀连接起来。

## 3. 第一版系统定位

### 3.1 正式定位

Energy-Maintenance 第一版定位为：

```text
面向华为与阳光电源光伏逆变器的多模态大模型检修知识检索、故障诊断与标准化作业辅助系统。
```

该定位包含五个关键词：

```text
1. 华为与阳光电源：明确厂家范围。
2. 光伏逆变器：明确设备类型。
3. 多模态：支持文本、图片、型号、告警码等输入。
4. 大模型：支持本地小模型与云端模型统一接入。
5. 作业辅助：不仅问答，还要支持 SOP、任务和维修记录。
```

### 3.2 第一版支持范围

第一版支持厂家：

```text
huawei：华为
sungrow：阳光电源
```

第一版支持产品系列：

```text
SUN2000
FusionSolar
SG
```

第一版支持设备类型：

```text
pv_inverter：光伏逆变器
```

### 3.3 第一版不覆盖范围

第一版不作为主线支持：

```text
1. 摩托车发动机维修。
2. 汽车制造全产线设备。
3. 储能电池系统。
4. 箱式变压器。
5. 电力巡检机器人。
6. 通用工业设备检修。
7. 所有新能源设备。
8. 完整电站 SCADA 或 IoT 监控平台。
```

说明：这些方向可以作为系统后续扩展能力，但不能进入第一版核心开发范围，避免系统失焦。

## 4. 建设目标

### 4.1 总体目标

构建一个可部署、可演示、可持续扩展的设备检修智能辅助系统，使一线运维人员能够围绕具体设备快速完成资料检索、故障诊断、作业指引、任务处理和维修经验沉淀。

### 4.2 业务目标

```text
1. 降低新手检修人员查找资料和理解告警的门槛。
2. 提高设备故障响应效率。
3. 规范检修步骤，降低误操作风险。
4. 将历史故障和维修经验沉淀为设备履历。
5. 将一线经验和模型修正结果纳入知识库。
6. 支撑后续同类故障复发判断和经验复用。
```

### 4.3 技术目标

```text
1. 使用 FastAPI + Vue3 + PostgreSQL 构建 B/S 系统。
2. 支持 LoongArch + 银河麒麟原生部署。
3. 支持本地小模型服务接入。
4. 支持云端大模型作为备用或增强。
5. 建立统一 Model Gateway 管理模型调用。
6. 支持多模态输入，包括文本、故障图片、设备型号、告警码。
7. 支持知识资料解析、切片、检索和来源追溯。
8. 支持后续扩展 embedding、OCR、图像识别和知识图谱。
```

## 6. 核心业务对象

### 6.1 设备台账

设备台账是系统的核心数据基础。系统需要知道每台逆变器是谁、在哪里、什么型号、历史上出过什么问题。

设备台账至少包含：

```text
设备编号
设备名称
厂家
产品系列
设备型号
设备类型
所属电站
安装位置
投运日期
当前运行状态
最近故障时间
最近检修时间
累计故障次数
累计检修次数
备注
```

### 6.2 设备维修履历

设备维修履历用于记录每台设备的历史问题、诊断过程、维修措施和复检结果。

必须支持回答：

```text
这台设备之前是否出过类似问题？
上次是什么告警？
上次如何处理？
是否维修完成？
这次是否可能是旧问题复发？
是否可能诱发了新的问题？
同型号设备是否有类似历史故障？
```

维修履历至少包含：

```text
设备 ID
故障时间
故障类型
告警码
故障现象
诊断结论
维修措施
更换部件
处理结果
是否复发
复检结果
维修人员
完成时间
关联任务
关联诊断记录
关联问答记录
```

### 6.3 多模态知识文档

知识来源包括：

```text
官方公开手册
安装维护手册
用户手册
告警代码说明
运维指导资料
巡检规范
检修 SOP
故障案例
一线经验总结
模型输出修正内容
人工标注内容
故障图片说明
```

### 6.4 知识切片

知识切片是系统检索和大模型回答的基础。

要求：

```text
1. 由真实资料解析生成。
2. 保留来源文档、页码、章节、厂家、产品系列和设备类型。
3. 支持检索问答时作为 references 返回。
```

### 6.5 检修问答记录

记录用户问题、模型回答、检索片段、来源依据和 trace_id。

### 6.6 故障诊断记录

记录故障描述、故障类型、可能原因、排查步骤、安全提示、推荐措施和 trace_id。

### 6.7 标准化作业指引 SOP

SOP 用于将诊断结果转化为可执行的检修流程。

SOP 需要体现：

```text
作业前准备
安全确认
停机与隔离要求
故障确认
检查步骤
维修步骤
复检步骤
恢复运行
记录归档
合规提醒
```

### 6.8 检修任务与维修记录

检修任务负责过程管理，维修记录负责沉淀结果。

### 6.9 知识投稿与审核

一线人员上传经验总结、维修案例、图片说明或模型修正建议后，应进入审核流程，专家审核通过后才能纳入正式知识库。

### 6.10 模型调用记录

所有本地模型和云端模型调用都应记录到 model_call_logs，用于审计、性能评估和后续优化。

```text

```

### FR-004 设备维修履历管理

系统应支持记录设备每次故障和维修处理情况。

功能要求：

```text
1. 从检修任务生成维修履历。
2. 手动补录历史维修记录。
3. 查看单设备维修时间线。
4. 标记是否复发。
5. 记录故障原因、维修措施、复检结果。
6. 关联诊断记录、问答记录、任务记录。
```

验收标准：

```text
1. 某台设备重复故障时，系统能展示历史相似问题。
2. 维修完成后形成可追溯履历。
3. 履历与 device_id 绑定。
```

### FR-006 多模态智能检索

系统应支持基于文本、告警码、设备型号和图片说明的检索。

第一版检索输入：

```text
query
device_id
manufacturer
product_series
model
alarm_code
fault_image
image_description
document_type
top_k
```

第一版检索方式：

```text
1. 关键词检索。
2. 告警码精确匹配。
3. 厂家/系列/型号过滤。
4. 图片人工描述参与文本检索。
5. 后续预留 embedding 和图像向量检索。
```

验收标准：

```text
1. 检索结果必须来自真实知识切片。
2. 返回来源文档、章节、页码、chunk_index。
3. 不得编造不存在的 references。
```

### FR-007 大模型检修问答

系统应支持本地小模型和云端模型统一接入。

模型服务架构：

```text
Model Gateway
    ├── Local Model Adapter
    └── Cloud Model Adapter
```

问答流程：

```text
用户问题
    ↓
检索知识切片
    ↓
构造上下文
    ↓
调用本地模型或云端模型
    ↓
生成答案、步骤和安全提示
    ↓
保存 qa_records 和 model_call_logs
```

模型策略：

```text
1. 默认优先本地小模型。
2. 本地模型不可用时，可切换云端模型。
```

```text

4. 模型调用记录写入 model_call_logs。
```

## 16. 风险分析

### 16.1 功能范围偏大风险

当前第一版功能较完整，开发工作量明显增加。应通过分阶段开发控制风险。

### 16.2 本地模型适配风险

LoongArch + 8GB 内存环境可能无法流畅运行较大模型。应优先实现 Model Gateway 和小模型适配接口，模型规模根据环境调整。

### 16.3 多模态实现风险

第一版多模态应先落地“图片上传 + 图片说明 + OCR 可选 + 关联检索”，避免一开始追求复杂视觉大模型。

### 16.4 资料获取风险

华为和阳光电源完整内部维修资料难以获取，应以公开资料、样例资料、一线经验和用户上传资料构建资料库。

## Task 02A 需求分析补充确认

本节用于确认新版功能边界，后续数据库、接口和前端实现应与本节保持一致。

### A. P0 功能结论

- Device ledger：P0。第一版本必须具备华为、阳光电源光伏逆变器设备台账。
- Device maintenance history：P0。检索和诊断需要可关联设备历史检修记录。
- Permission system：P0。第一版本至少需要基础用户、角色、状态和密码哈希字段预留。
- Local small model architecture：P0。通过 Model Gateway 预留本地 llama.cpp + GGUF 路线。
- Fault image upload：P0 multimodal。第一版本支持故障图片上传与描述记录。
- SOP：P0。需要 SOP 模板和执行记录支撑现场作业辅助。
- Knowledge contribution / review / model output correction：P0。知识贡献、审核和模型输出纠错需要形成闭环。
- Trace center：P0。QA、诊断、任务、模型调用和操作日志需要可追溯。

### B. P1 或后续增强结论

- OCRService reserved：P0 预留抽象；具体 OCR 引擎选择为 P1，需经过 LoongArch + Kylin 验证。
- pgvector：P1，不阻塞 P0。
- embedding hybrid retrieval：P1，不阻塞 PostgreSQL 关键词检索闭环。
- real LLM generation：后续增强，不作为 P0 闭环前置条件。

### C. 第一版本业务边界

第一版本聚焦华为、阳光电源光伏逆变器，不扩展为通用新能源设备平台。新增业务字段、接口参数和前端选项应围绕 manufacturer、product_series、model、device_type、fault_type、alarm_code、trace_id、references 设计。
